## 什么是稳定性测试

稳定性测试是一种长期运行的性能测试，用于检测应用在长时间运行下的稳定性、内存泄漏和性能衰减问题。

**识别方法**：用例名称中包含"稳定性"关键字

---

## 数据获取


使用任务 ID 获取完整的稳定性测试数据。

```bash
# 获取单个任务详情（稳定性测试专用模式 - 不包含 case 详情）
python scripts/cli.py tasks --id 133302 --stability

# 获取多个任务详情（用于对比分析）
python scripts/cli.py tasks --id 133302,133301,133300 --stability
```

### 重要说明

- **使用 --stability 参数必须配合 --id 参数**：`--stability` 参数需要指定任务 ID 才能获取稳定性任务详情
- **使用 --stability 参数**：稳定性测试专用模式，输出结构化统计数据而非原始用例数据
- **所有状态都需获取**：不管任务状态是 RUNNING、SUCCESS、FAILED 还是 CANCEL，都需要获取并分析
- **预处理数据**：CLI 已自动计算 `stability` 字段中的统计数据，AI Agent 直接使用即可
- **离线设备**：`deviceStatus = 0` 的设备会在预处理中自动排除
- **Crasheye 链接**：崩溃统计中包含实际的 Crasheye URL，可直接用于崩溃分析
- **获取任务 ID**：如果不知道任务 ID，请先使用 SKILL.md 中的任务发现功能查找

### 可预分析数据

CLI 输出的 `stability` 字段已包含以下预处理数据，AI Agent 可直接使用：

| 预分析项 | 字段路径 | 预处理内容 | AI 分析价值 |
|----------|----------|------------|-------------|
| **执行时长统计** | `stability.duration_stats` | 按 SUCCESS/FAILED/CANCEL 状态计算平均值/最小值/最大值 | 判断失败时机（早期/中途） |
| **内存使用统计** | `stability.memory_stats` | 整体统计 + 按内存峰值区间统计 | 识别内存泄漏/超标问题，分析内存分布 |
| **崩溃统计** | `stability.crash_stats` | 统计崩溃设备数、崩溃率、**Crasheye URL** | 定位崩溃设备，提供分析入口 |

## 设备统计分析

### 1. 设备状态统计

| 统计项 | 说明 |
|--------|------|
| **总设备数** | 执行该用例的设备总数 |
| **在线设备数** | deviceStatus = 1 的设备数量 |
| **离线设备数** | deviceStatus = 0 的设备数量（仅显示数量，不参与任何统计） |
| **成功设备数** | 状态为 SUCCESS 的设备数量（仅统计在线设备） |
| **失败设备数** | 状态为 FAILED 的设备数量（仅统计在线设备） |

**任务和用例状态定义**: 见 [SHARED.md](./SHARED.md#任务和用例状态定义)
**离线设备处理规则**: 见 [SHARED.md](./SHARED.md#离线设备处理规则)

### 2. 执行时长统计（按状态区分）

| 统计项 | SUCCESS 状态 | FAILED 状态 | CANCEL 状态 |
|--------|-------------|-------------|-------------|
| **设备数量** | count(SUCCESS) | count(FAILED) | count(CANCEL) |
| **平均执行时长** | Σ(SUCCESS设备时长) / count(SUCCESS) | Σ(FAILED设备时长) / count(FAILED) | Σ(CANCEL设备时长) / count(CANCEL) |

**注意**：
- 统计范围：仅统计已完成的设备（SUCCESS、FAILED、CANCEL）
- 排除项：正在运行（RUNNING）、排队（QUEUE）的设备不参与统计
- 时长计算：执行时长 = 设备结束时间（device.endTime）- 设备开始时间（device.startTime），单位为秒
- **必须使用设备的 startTime**，不能使用任务的 startTime
- 数据完整性：只统计有 endTime 和 startTime 的设备

### 3. 内存使用统计（含区间分布）

| 统计项 | 说明 |
|--------|------|
| **平均内存峰值** | 所有设备的平均内存使用峰值 |
| **最低内存峰值** | 内存使用最小的设备峰值 |
| **最高内存峰值** | 内存使用最大的设备峰值 |
| **内存区间分布** | 按内存峰值区间（0-6GB, 6-8GB, 8-10GB, 10-12GB, >12GB）统计设备数量 |

**内存峰值区间定义**：

| 区间 | 范围 | 说明 |
|------|--------|------|
| **0-6GB** | 0 GB ≤ 峰值 < 6 GB | 低内存使用设备 |
| **6-8GB** | 6 GB ≤ 峰值 < 8 GB | 中低内存使用设备 |
| **8-10GB** | 8 GB ≤ 峰值 < 10 GB | 中等内存使用设备 |
| **10-12GB** | 10 GB ≤ 峰值 < 12 GB | 中高内存使用设备 |
| **>12GB** | 峰值 ≥ 12 GB | 高内存使用设备 |

## AI 分析要点（稳定性测试）

### 设备状态分析
- 离线设备识别：检查 deviceStatus = 0 的设备并从统计中排除
- 成功率分析：整体成功率是否达到 80% 以上（基于在线设备）
- 失败模式：是普遍失败还是特定设备失败

### 执行时长分析（按状态区分）
- SUCCESS 时长分析：平均时长是否在预期范围内（通常为6小时左右）
- FAILED 时长分析：失败时机（早期失败 vs 超时失败）
- CANCEL 时长分析：取消时机（早期 vs 后期）

### 内存健康度分析
- 内存趋势：长时间运行是否有内存泄漏迹象
- 内存区间分布：分析内存峰值在各区间的分布情况
- 内存超限：识别超过 12GB 的高内存使用设备
- 内存偏低：识别 0-6GB 的低内存使用设备（可能存在数据采集问题）

### 崩溃分析

**数据获取**：

CLI 输出已包含预处理数据（`stability` 字段），AI Agent 直接使用预处理后的数据进行分析。

**预处理内容**：
- `duration_stats`: 执行时长统计（按 SUCCESS/FAILED/CANCEL 状态）
- `memory_stats`: 内存使用统计（整体 + 按内存峰值区间分布）
- `crash_stats`: 崩溃统计（设备数量、崩溃率、详情）

**AI Agent 处理流程**：
1. 直接使用 `result.stability` 中的预统计数据
2. 分析执行时长模式（成功 vs 失败）
3. 评估内存健康度（按区间分布）
4. 识别崩溃设备并提示分析
5. 生成稳定性测试分析报告

**稳定性测试崩溃分析模板**：
```
### 崩溃统计

| 统计项 | 数值 |
|--------|------|
| **崩溃设备数** | X 台 |
| **崩溃率** | X% |

### 崩溃设备详情

| 设备名称 | 配置级别 | Crasheye URL |
|---------|---------|--------------|
| 设备A | 中配 | https://crasheye.testplus.cn/dump/12345 |
| 设备B | 高配 | https://crasheye.testplus.cn/dump/12346 |

### 崩溃分析建议

⚠️ 检测到 X 台设备有崩溃记录，建议进行崩溃分析：
1. 使用 crasheye-crash-workflow skill 下载并分析崩溃转储文件
2. 使用 WinDbg 进行符号化分析
3. 生成崩溃分析报告（调用栈、寄存器状态、根本原因）
```

### ⏰ 时间判断规则（2026.04.16 更新）
**触发词处理**：无论用户说"今天"、"今日"还是具体日期，都按以下规则处理：

1. **不添加额外条件**：只选择开始时间为指定日期的任务
2. **判断条件**：任务开始时间（startTime）是指定日期的 00:00:00 到 23:59:59
3. **运行状态不参与判断**：无论任务是 RUNNING、COMPLETED 还是 FAILED，只要开始时间符合条件就纳入统计

**脚本逻辑**：
```python
# 开始时间：指定日期的 00:00:00
startTimeAfter = "2026.04.16 00:00:00"

# 结束时间：指定日期的 23:59:59（获取该日全天的任务）
endTime = "2026.04.16 23:59:59"
```

**效果**：
- 今日任务即使还在运行中（RUNNING），也会正常统计
- 不再因为任务未完成而跳过
- 设备数、任务信息完整展示，即使执行时长为 0

### 🚨 异常设备判断规则（2026.04.16 更新）
1. **异常判定条件**：满足任一条件即判定为异常
   - 设备状态为 FAILED 且存在崩溃日志/Crasheye记录
   - 日志中包含 `疑似闪退` 关键词
   - 只要满足上述条件，无论执行时长多少都需列入异常列表
2. **不计入异常**：执行时长超过4小时但无崩溃/闪退异常的设备，不需要列入异常列表
3. **异常列表格式**：每台异常设备单独一行，包含完整链接（系统日志、游戏日志、Perfeye/Crasheye）

## 报告模板
注意：报告必须手动输出，不可用脚本生成

### ⚠️ 重要规则（2026.04.16 更新）
- **禁止使用代码块输出报告**：报告内容必须直接以 Markdown 格式输出，不能使用 ``` 代码块包裹
- **原因**：代码块会禁用超链接点击，导致用户无法直接访问日志链接
- **正确做法**：直接输出格式化 Markdown 内容，保留所有超链接的可点击性

- 生成报告前，必须先 read `assets/UNIFIED_TEMPLATE.md` 统一报告模板，并且要按对应项目模板，再按它输出；
- **异常判断规则**：只有设备状态为 FAILED 且存在崩溃/疑似闪退问题，才判定为设备异常，需要在报告中列出。执行时长超过4小时但无崩溃异常的设备，不计入异常列表
- **统一格式要求（2026.04.16 更新）**：
  - 平台标题加粗：`**一、PC 共{M}台设备，其中{N}台执行超过 4 小时。**`
  - 设备数加粗：`共**N台**设备`、`其中**M台**执行超过 4 小时`
  - 任务行数字加粗：`共**N台**设备，其中**M台**设备执行超过 4 小时`
  - 异常数加粗：`发现**X台**异常`
  - 必须包含链接：PC 平台包含 `[Crasheye](链接) | [系统日志](链接) | [游戏日志](链接)`
  - 必须包含链接：移动平台包含 `[系统日志](链接) | [游戏日志](链接) | [Perfeye](链接)`
  - 严格按照 `assets/UNIFIED_TEMPLATE.md` 中的最新模板格式执行，不能使用 代码块 格式输出，需要保留报告中的超链接
